### Topic

Past, Current and Future of Big Data

### Summary

This academic paper reviewed the evolution of big data technology — from early on-premise systems like Apache Hadoop to modern cloud-based Data Lakehouse architectures. It covered how industries moved away from managing hundreds of physical servers due to cost and scalability issues, and how platforms like Azure now provide the infrastructure needed for real-time streaming and large-scale analytics. The paper also explored tools like Apache Kafka for stream processing and discussed future directions in data engineering.

### Reflection

Writing this paper helped me understand the bigger picture of where data engineering comes from and where it is heading. Before this, I knew individual tools but had not connected them into a historical narrative. Understanding why Hadoop was built, and why the industry eventually moved toward cloud and streaming architectures, gave me a much stronger foundation for everything else in this course. The TurnItIn submission also pushed me to be more careful about how I present and cite information.

## 🤖 Assignment 3 — Tutorial: AI Algorithm CNN

> 📁 Folder: [`Assignment3/`](./Assignment3)

### Topic

Image classification using Convolutional Neural Networks (CNN) on the CIFAR-10 dataset.

### Summary

This tutorial covered building and enhancing a CNN model for image classification on CIFAR-10 — a dataset of 60,000 images across 10 classes (airplane, automobile, bird, cat, deer, dog, frog, horse, ship, truck). The baseline CNN achieved 77.53% accuracy. My section focused on improving it by adding data augmentation (RandomFlip, RandomTranslation, RandomRotation, RandomZoom), a deeper three-block architecture (32→64→128 filters), Batch Normalisation, and replacing the Flatten+Dense(512) layer with Global Average Pooling. The enhanced model achieved 80.45% accuracy with 75.5% fewer parameters (306,602 vs 1,251,858).

### Reflection

This was the assignment where I finally understood what "overfitting" really means in practice — not just as a definition, but as something I could see in the training curves. The baseline model started diverging at epoch 6, and the fixes we applied (data augmentation, BatchNorm, GAP) each addressed a specific cause. Achieving better accuracy with 75% fewer parameters was a satisfying result — it showed that good architecture choices matter more than just making a model bigger. I also got a lot more comfortable reading and interpreting training curves, confusion matrices, and per-class accuracy.

> **Suggestion for Improvement:** Running the model on GPU would allow testing more configurations in less time, which would make the exploration phase more productive.

## ⚡ Assignment 4 — Tutorial: Generative AI

> 📁 Folder: [`Assignment4/`](./Assignment4)

### Topic

Building an AI-assisted ETL data pipeline using Generative AI (Claude).

### Summary

This tutorial explored using Claude (a generative AI model) to generate Python code for each stage of a 4-stage ETL pipeline applied to the Dirty Cafe Sales dataset from Kaggle — a synthetic dataset with intentional data quality issues: missing values, ERROR/UNKNOWN placeholders, inconsistent date formats, and duplicate rows. The goal was to see how AI can accelerate data engineering tasks while still requiring human oversight for validation and correctness.

### Reflection

This tutorial changed how I think about AI as a tool. Using Claude to generate pipeline code was faster than writing it from scratch, but it was not automatic — we still had to understand what the code was doing, catch errors, and validate the output at each stage. The "AI assists, human verifies" workflow felt very realistic. It also made me more aware of data quality as a serious problem: real datasets are messy, and a pipeline needs to handle edge cases, not just the happy path. I came away from this more confident in both using AI tools and in my own ability to evaluate the code they produce.

> **Suggestion for Improvement:** Comparing outputs from multiple AI models (e.g., Claude vs GPT vs Gemini) for the same pipeline tasks would add an interesting benchmarking dimension to the assignment.

## 🏗️ Project Industry (PPG)

> 📁 Folder: [`ProjectIndustry/`](./ProjectIndustry)

### Project Overview

End-to-end automated data engineering pipeline for **PPG Coatings (M) Sdn Bhd** — a real industry client. The system addresses inventory obsolescence and detects manual recording issues for raw material losses across departments.

### Technologies Used

Azure Data Factory · Azure Data Lake Storage Gen2 · Azure Databricks · PySpark · Azure Synapse Analytics · Medallion Architecture · Galaxy Schema

### My Contributions

As part of a 7-member team, I contributed to the data pipeline design and implementation. The project used the Waterfall methodology and Medallion Architecture with four layers — Bronze (raw ingestion), Silver (cleaning), Gold (business calculations), and Curated (final output). Azure Data Factory orchestrated extraction of six CSV operational data files into ADLS Gen2. PySpark on Azure Databricks handled data cleaning and calculation of 12-month raw material usage indicators. Azure Synapse Analytics hosted the Galaxy Schema data model for analysis.

### Reflection

The PPG project was the most realistic experience of the whole semester. Working with a real client's data — six actual CSV files from a manufacturing company — was very different from working with clean tutorial datasets. The inventory classification logic (Recoverable Assets, Magna Carta, dead inventory, expired materials) had to be precise because it directly affects the company's financial provisions. I learned how to translate a business problem into a data engineering solution, and how the tools we studied (Azure, PySpark, Medallion Architecture) fit together in a production-style system. Collaborating in a team of seven also taught me the importance of version control, clear task division, and consistent documentation.

> **Suggestion for Improvement:** Having earlier access to the client's data dictionary would have saved significant time during the Bronze-to-Silver transformation stage.

## 💭 Overall Reflection

### What I Gained This Semester

SECP3843 was the most intensive and rewarding course of my degree so far. Across the assignments, industry engagement, and projects, I got hands-on experience with the full modern data engineering stack — cloud infrastructure (Azure), big data processing (PySpark, Databricks), AI and ML (CNN, Generative AI), and real client work (PPG).

The combination of tutorials, industry talk, industry visit, and the PPG project gave me a 360° view of the field: theory, tools, professional context, and real-world application in a single semester.

### Key Takeaways

- **Cloud is the default** — every serious data pipeline today runs on cloud infrastructure. Azure became familiar territory after this semester.
- **Architecture decisions matter more than tool choices** — the CNN assignment showed that a smarter design (GAP, BatchNorm) beats a bigger model. The same principle applies to data pipelines.
- **AI assists, humans verify** — the Generative AI tutorial showed me how to use AI productively without blindly trusting its output.
- **Real data is messy** — both the PPG project and the Generative AI cafe dataset reinforced that data quality is a core engineering problem, not a cleanup step.
- **Industry context changes everything** — the visit and PPG project made abstract concepts concrete.

### Looking Forward

I plan to pursue Azure certifications (AZ-900, DP-203) and deepen my PySpark skills. I also want to keep building on the CNN and Generative AI work from this semester as part of my personal portfolio.
